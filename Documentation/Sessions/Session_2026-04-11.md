## Session Summary

Two-part working session (split across context boundaries). The first part completed the excision of all spurious `OTr_zSetOK(1)` calls from method bodies, begun in a prior session. The second and dominant part implemented the shadow-type key scheme — a solution to the fundamental type-disambiguation problem: 4D's object model collapses Pointer and (on pre-v19 R2) BLOB to `Is text` under `OB Get type`, making it impossible for `OTr_zMapType` and therefore `OTr_ItemType` to report the correct OT type without extra information. The shadow scheme stores the OT type constant in a sibling key suffixed `$type` (e.g. `{"ptr": "->myVar", "ptr$type": 23}`), guaranteeing correctness without any sentinel-prefix scheme or runtime value inspection.

Alongside the shadow scheme, longstanding gaps in `OTr_CopyItem` and `OTr_RenameItem` (missing Date and Time branches) and in all three property-reporting methods (missing Date and Time data-size cases) were repaired. The array-type branch of `OTr_zMapType`, which previously returned a hard-coded `114` (OT Object) for all sub-objects regardless of whether they were OTr array containers, was rewritten to descend into the sub-object and consult the `arrayType` property written by `OTr_PutArray`. `OTr_uMapType` was extended with a full 4D → OT array-element-type table to support that branch.

### 2. Shadow-type key helpers — OTr_zShadowKey and OTr_zIsShadowKey

Two new private helpers establish the shadow-key convention.

**`OTr_zShadowKey($leafKey_t)`** appends `"$type"` to the leaf key and returns the result. The `$` character cannot appear in a user-defined 4D identifier, so the suffix is a guaranteed non-collision namespace.

**`OTr_zIsShadowKey($key_t)`** returns `True` if the key ends with `"$type"`. Used as a skip-filter in the property enumerators to hide shadow keys from callers.

The shadow's stored value is the **OT** type constant (not the 4D native constant), keeping it aligned with `OTr_ItemType`'s public contract.

**Files created:**
- `Project/Sources/Methods/OTr_zShadowKey.4dm`
- `Project/Sources/Methods/OTr_zIsShadowKey.4dm`

## The Shadow-Type Key Scheme — Design Summary

The core problem: `OB Get type` returns `Is text` for three distinct kinds of stored value — ordinary user text (OT type 112), a text-encoded Pointer (OT type 23), and a text-encoded BLOB on pre-v19 R2 (OT type 30). `OTr_zMapType` has no way to distinguish them from the type tag alone.

The solution: when a value must be stored as text for representation reasons, write a sibling key whose name is the leaf key plus `"$type"`, storing the integer OT type constant.

```
{"myPtr": "->Table_1_1", "myPtr$type": 23}
```

Reading back: `OB Get type` returns `Is text`; `OTr_zMapType` checks whether `leafKey$type` is defined; if so, uses that value as the OT type. If not, assumes ordinary text → OT Character (112).

The `$` character is not a valid character in a 4D developer identifier, so the `$type` suffix cannot collide with any user-chosen key. Shadow keys are invisible to all enumeration methods and are moved, copied, or deleted automatically alongside their primary leaf.

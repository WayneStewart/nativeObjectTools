# Native Object Tools — Release Process

This document governs the release process for Native Object Tools. It is the canonical source of truth. The GitHub Actions workflow (`.github/workflows/release.yml`), the 4D release project (`Release/Project/`), and the `buildApp.4DSettings` files in `Release/BuildSettings/` implement what is described here. If the code and this document disagree, this document is correct and the code should be fixed.

## Purpose

Native Object Tools ships as two components that differ only in public API naming:

- **OTr_** style — native 4D prefix (e.g. `OTr_New`, `OTr_Pack`)
- **OT** style — legacy plugin-compatible naming (e.g. `OT New`, `OT Pack`)

Both components are produced from the same source tree. Users install whichever style suits their codebase. A single installer (`.dmg` for macOS, `.zip` for Windows) carries both components side by side.

Three installers are produced per release, one per supported 4D LTS version (19, 20, 21).

## Architecture in one sentence

A push to a release tag on `Echidna` triggers a GitHub Actions workflow on a self-hosted macOS runner, which drives 4D headlessly through a sequence of stages (stage, compile, sign, package, notarise, staple, zip, publish), writing a sentinel file at each stage so the workflow can detect and report failure.

## Branching model

Three branches exist. Only one is worked on directly.

- **`Echidna`** — the only development branch. All work happens here: code, tests, documentation, the release tool itself, the GitHub Actions workflow. It uses `OTr_` prefixes and carries all testing scaffolding, the Codex web-testing bridge, all draft documentation, and development-only resources.

- **`Koala`** — release record in `OTr_` state. Generated by the workflow by stripping Echidna of development-only material. Never checked out for editing. Force-pushed by the workflow at each release.

- **`Platypus`** — release record in `OT` state. Generated from Koala's output by running the Renamer in forward mode. Never checked out for editing. Force-pushed by the workflow at each release.

**Work flows only one way:** Echidna → Koala → Platypus. Changes are never merged back from release branches. The release branches exist so that `git log Koala` and `git log Platypus` give per-variant release history; they are not places where development happens.

## The dispatch pattern (learned from Roadshow)

One 4D project method dispatches all 4D-native work, driven by the `--user-param` flag.

`OnStartup` reads the user-param, splits it on `;` into a list of stage names, and dispatches each to a method. On any failure it writes a sentinel file and calls `QUIT 4D`. The shell step in the workflow reads the sentinel's first line and exits non-zero on failure.

The stage method signature is standardised: each stage writes its sentinel to a predictable path in the current working directory and returns `True`/`False` to indicate whether the next stage should proceed.

Sentinel file contract:

- Line 1: `"<stage> passed"` or `"<stage> failed"`.
- Remaining lines: structured diagnostic output (compile errors, notarisation log excerpt, etc.).

The workflow reads `head -n 1 <sentinel>.txt` and fails the job if line 1 does not end with `" passed"`.

Stage method naming: `Release_stageKoala`, `Release_compile`, `Release_build`, etc. — in the 4D release project under `Release/Project/Sources/Methods/`.

## The stages

Named pipeline stages, in order. Each is implemented once in 4D and invoked from the workflow with the appropriate user-param.

1. **`stageKoala`** — copies the Echidna checkout into `staging-koala/`, deletes paths listed in `Release/exclusions.md`, deletes `Compiler_Prototyping.4dm`, `git init`s the staging directory. Sentinel: `stage-koala.txt`.

2. **`stagePlatypus`** — copies `staging-koala/` into `staging-platypus/`, invokes the existing Renamer (`Renaming/`) in forward mode against the Platypus copy. Sentinel: `stage-platypus.txt`.

3. **`compile`** — invoked six times (three 4D versions × two variants). Uses `Compile project` with the appropriate `buildApp-NN-VARIANT.4DSettings` and writes errors from the `errors.query("isError == :1"; True)` result to the sentinel. Sentinel: `compile-NN-VARIANT.txt`.

4. **`build`** — invoked six times. Uses `BUILD APPLICATION` against the same `buildApp-NN-VARIANT.4DSettings`, emitting `Native Object Tools.4dbase` into `build-output/NN/VARIANT/`. Sentinel: `build-NN-VARIANT.txt`.

5. **`sign`** — not a 4D stage. A shell step runs `Release/Scripts/notarise.sh sign` which loops over the six built components and runs `codesign --remove-signature` then `codesign --sign "Developer ID Application: ..."` on each `lib4d-arm64.dylib`. Sentinel: `sign.txt`.

6. **`package`** — shell step. For each 4D version, edits the `dual-component.dmgcanvas` template to reference both that version's `OTr_` and `OT` built components, invokes `dmgcanvas`, and outputs `Object Tools NN LTS (V.V.V).dmg` to `Builds/Installers/`. Produces three DMGs. Sentinel: `package.txt`.

7. **`notarise`** — shell step. `xcrun notarytool submit ... --keychain-profile "OTr-Notary" --wait` against each DMG, followed by `xcrun stapler staple` on success and `xcrun stapler validate` as a check. Sentinel: `notarise.txt`. This is the slowest stage — typically 5–20 minutes per DMG.

8. **`zip`** — shell step. For each 4D version, produces a parallel `.zip` containing both `.4dbase` folders. No signing. Sentinel: `zip.txt`.

9. **`publish`** — shell step. Force-pushes `Koala` and `Platypus` branches to `origin`, tags Echidna with the version string, creates a GitHub Release, attaches the six artefacts (3 dmg + 3 zip) plus `checksums.txt`, appends an entry to `Release/releases.md` recording version + SHAs. Sentinel: `publish.txt`.

## Triggering a release

Two trigger modes, both defined in `.github/workflows/release.yml`:

**Tag-driven (normal release).** Push an annotated tag matching `v*.*.*` to Echidna. The workflow runs automatically. This is the expected path.

**Manual `workflow_dispatch` (ad-hoc).** Trigger from the GitHub UI with a version parameter. Useful for rebuilding an existing tag or producing a candidate from an arbitrary commit.

There is no local-only release path. Releases always go through the runner, so every release is reproducible and logged in GitHub Actions.

## The 4D command-line invocation

Each 4D stage is invoked via `open -W` (which blocks until the app exits). The exact form is:

```bash
open -W /Applications/4D/19/4D.app --args \
     --project "$STAGING/Project/NativeObjectTools.4DProject" \
     --create-data "$RUNNER_TEMP/data.4DD" \
     --user-param "compile;build" \
     --headless
```

- `-W` on `open` is essential — without it the shell returns immediately and the stage's sentinel file does not yet exist.
- `--headless` suppresses dialogs and the Design mode; all dialog calls are auto-answered.
- `--create-data` is used instead of relying on a shipped data file — the component has no data of its own, and `--dataless` causes `Compile project` to throw in some 4D versions.
- `--user-param` accepts semicolon-delimited stage names. The workflow batches efficiently: `compile;build` runs in a single 4D launch rather than two.

4D for each supported version is installed under `/Applications/4D/19`, `/Applications/4D/20`, `/Applications/4D/21` on the runner machine. The workflow selects the appropriate path by version number.

## The self-hosted runner

One self-hosted macOS runner is registered with the GitHub repository. Its workspace lives outside the repository, at `~/actions-runner/`.

**Host machine requirements:**

- macOS Sonoma or later (Apple Silicon recommended — notarisation of ARM64 components is straightforward there).
- 4D installed for each supported LTS version (19, 20, 21), at the paths above.
- DMG Canvas installed with CLI on `PATH`.
- Developer ID Application certificate in the login keychain.
- `notarytool` keychain profile `"OTr-Notary"` configured.
- GitHub Actions runner agent installed and registered.

**Why self-hosted, not GitHub-hosted:**

- 4D's licence and installer cannot be downloaded fresh on every run without storing a licensed installer URL in a secret and accepting the runtime cost.
- The Developer ID certificate must live on the signing machine; exporting it into a GitHub-hosted runner on every run is undesirable from a security posture.
- Notarisation is long-running (minutes); dedicated hardware is fine, billed CI minutes would be wasteful.

The runner does not need to be always-on — GitHub Actions queues jobs and waits for the runner to come online. Starting the runner an hour before cutting a release is sufficient.

## The exclusion manifest

`Release/exclusions.md` is a committed markdown file listing every path stripped from the Koala staging tree. It is the single source of truth for what is "development-only". When the question arises six months hence — *"why was this file removed from the release?"* — the answer is a line in that table, not a line in a workflow file.

Entries are classified:

- **Always strip** — deleted unconditionally (e.g. `____Test_*.4dm`, `Codex_*.4dm`, `WebFolder/`).
- **Keep only these** — for directories where most contents go but some stay, the rule is inverted (e.g. `Resources/` — keep only `InfoPlist.json` and `InfoPlist.strings`).
- **Conditional** — reserved for rules that depend on release type.

Changes to what gets stripped are made by editing `exclusions.md`. The `Release_stageKoala` method reads the file; it does not carry a hard-coded list.

## Development-only code lives in `Compiler_Prototyping.4dm`

All compiler declarations for test-only arrays, Codex bridge variables, and constants consumed only by tests are consolidated in `Compiler_Prototyping.4dm`. The `stageKoala` stage removes this single file and leaves the production compiler methods (`Compiler_ObjectToolsReplacement.4dm`, `Compiler_FCS.4dm`, and the `Compiler_Log*.4dm` family) untouched.

If during development a new declaration is added that supports only test or web-bridge code, it must go into `Compiler_Prototyping.4dm`. If it supports shipping functionality, it must go into one of the production compiler files. This is the one rule contributors must hold in their head; the compiler's file partition makes the rule self-enforcing in review.

## Image methods handle missing resources gracefully

Each `OTr_z_[Animal].4dm` method (one per image file in `Resources/images/`) returns an empty picture variable when its associated `.png` is absent from `Resources/images/`. This permits the release workflow to strip `Resources/images/` without also stripping the methods that reference it. The methods remain; the images do not.

The two InfoPlist files in `Resources/` (`InfoPlist.json`, `InfoPlist.strings`) are retained — they are required for the component to load.

## The `buildApp.4DSettings` files

Six `buildApp.4DSettings` files live in `Release/BuildSettings/`, one per (4D-version, variant) pair. Each is edited to target its specific staging directory and output location. They are committed configuration, not generated at runtime — edits require a commit, making changes visible in review.

Because 4D's BUILD APPLICATION looks for `buildApp.4DSettings` in the project's `Settings/` folder, the `stageKoala` and `stagePlatypus` stages copy the appropriate `buildApp-NN-VARIANT.4DSettings` into each staging tree's `Settings/` directory as `buildApp.4DSettings` before the compile stage runs.

## First-time setup on the runner

A checklist of things to configure once per runner machine:

- **Install 4D.** One installation for each supported LTS version, at `/Applications/4D/19`, `/Applications/4D/20`, `/Applications/4D/21`.
- **Install DMG Canvas** and verify `which dmgcanvas` resolves.
- **Import Developer ID certificate** into the login keychain.
- **Create the notarytool keychain profile:**
  ```
  xcrun notarytool store-credentials "OTr-Notary" \
       --apple-id "<your-apple-id>" \
       --team-id "H8FSU88B4T" \
       --password "<app-specific-password>"
  ```
- **Register the GitHub Actions runner agent.** Follow GitHub's "Add self-hosted runner" flow in the repository Settings → Actions → Runners. Install as a service so it survives reboots: `./svc.sh install && ./svc.sh start`.
- **Verify 4D runs headless.** A smoke test: `open -W /Applications/4D/20/4D.app --args --headless --version` should print the version and exit.

## Producing a release — six-step procedure

1. Ensure Echidna is clean and all intended changes are committed.
2. Bump the version in `Release/version.txt` and commit.
3. Tag: `git tag -a v1.0.0 -m "Release 1.0.0" && git push origin v1.0.0`.
4. The GitHub Actions workflow runs. Watch its progress in the Actions tab.
5. On completion, download the GitHub Release artefacts or verify locally by inspecting the force-pushed `Koala` and `Platypus` branches.
6. Note the new entry in `Release/releases.md` (the workflow appended it on the publish stage) and pull that commit.

The full pipeline typically takes 20–40 minutes, dominated by notarisation.

## Troubleshooting

When things fail six months from now, these are the first places to look:

- **Workflow fails at `stageKoala`.** The exclusion manifest may reference a path that no longer exists, or `Compiler_Prototyping.4dm` is missing. Read `stage-koala.txt` from the workflow logs.
- **Workflow fails at `compile`.** Read the sentinel — the error message includes the method, line, and `Compile project` diagnostic. Usually a genuine compile error in Echidna that was not caught before pushing.
- **Workflow fails at `notarise`.** Read `notarise.txt`, then on the runner machine run `xcrun notarytool log <submission-id> --keychain-profile "OTr-Notary"` for the structured Apple response. Most rejections are unsigned Mach-Os or missing hardened runtime flags — check that the `sign` stage covered *both* staged components.
- **Runner offline.** The job sits queued. Start the runner service on the host machine: `cd ~/actions-runner && ./svc.sh start`.
- **DMG Canvas CLI not found.** Reinstall DMG Canvas and confirm `which dmgcanvas` succeeds in the runner user's shell environment (note: environment may differ from your interactive shell).
- **Force-push rejected on Koala or Platypus.** Someone has manually pushed to those branches. They should not. Investigate before force-pushing again; likely a mis-targeted commit from a local checkout.

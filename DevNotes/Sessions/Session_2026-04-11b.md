### 1. OTr_ItemCount — shadow-key exclusion

Phase 10a revealed that `OTr_ItemCount` was returning 14 where `OT ItemCount` returned 12 for the same object. The two extra items were the shadow keys (`ptr$type` and `blob$type`) introduced by the shadow-type key scheme. The `For each` loops in `OTr_ItemCount` were already filtering the `__otr_` internal prefix, but the `$type`-suffixed shadow keys were not covered.

Fixed by adding `& (Not(OTr_zIsShadowKey($thisKey_t)))` to the guard in both For each loops (the top-level path and the embedded-tag path).

**File:** `Project/Sources/Methods/OTr_ItemCount.4dm`

### 2. OTr_GetVariable — coercion-on-mismatch divergence documented

Phase 10a identified that `OTr_GetVariable` sets OK=0 on a type mismatch (correct) but returns the 4D-coerced value rather than the type's empty/default value (OT returns the empty value). This arises because `OB Get` with a type argument performs coercion when 4D can compute one.

Rather than suppressing the coercion — which would require an explicit `OB Get type` pre-check before every `OB Get` call in the method — the divergence was instead documented in the method header. Both implementations set OK=0 on mismatch, so callers that check OK will behave correctly regardless of the returned value.

**File:** `Project/Sources/Methods/OTr_GetVariable.4dm`

### 4. OTr_GetArray — typed OB Get to prevent type-mismatch crash

`OTr_GetArray` was calling `OB Get($parent_o; $leafKey_t)` without a type argument. When the tag held a scalar (Long etc.) rather than an array object, the untyped `OB Get` returned a Long integer, which could not be assigned to the `$arrayObj_o : Object` variable, producing error 54 at runtime.

Fixed: `OB Get:C1224($parent_o; $leafKey_t; Is object:K8:27)`. When the tag holds a scalar, `OB Get` with `Is object` returns Null; `OTr_zArrayType(Null)` returns -1, triggering the existing graceful error path.

The same untyped `OB Get → Object` pattern was then identified by grep across all array-operation methods and fixed proactively in six further methods:

- `OTr_DeleteElement.4dm`
- `OTr_FindInArray.4dm`
- `OTr_InsertElement.4dm`
- `OTr_SizeOfArray.4dm`
- `OTr_zSortValidatePair.4dm`
- `OTr_ArrayType.4dm`
- `OTr_ResizeArray.4dm`

### 6. OTr_GetPicture — OB Get type pre-check

`OB Get` with `Is picture` on a non-picture property throws hard error 64 ("A picture field or variable was expected") rather than coercing silently. Unlike the other getters, supplying the type argument is not sufficient — the call itself is fatal. Fixed with an `OB Get type` pre-check:

```4d
If (OB Get type:C1230($parent_o; $leafKey_t)=Is picture:K8:10)
    $result_pic:=OB Get:C1224($parent_o; $leafKey_t; Is picture:K8:10)
Else
    OTr_zError("Type mismatch: tag does not reference a Picture"; Current method name:C684)
End if
```

### 8. Scalar Put methods — type-consistency guard

The core defect behind several discrepancies: all six scalar Put methods called `OB SET` unconditionally after resolving the path, with no check on whether a same-key item of a different type already existed. OT's documented behaviour is to refuse the write, generate an error, and set OK=0 when the stored type does not match the put type.

Added a type-consistency guard to each:

```4d
If (OB Is defined($parent_o; $leafKey_t) \
    & (OB Get type($parent_o; $leafKey_t)#Is <type>:K8:N))
    OTr_zError("Type mismatch"; Current method name)
Else
    OB SET($parent_o; $leafKey_t; $inValue_x)
End if
```

| Method | Guard type constant | Note |
|---|---|---|
| `OTr_PutBoolean.4dm` | `Is boolean:K8:9` | |
| `OTr_PutDate.4dm` | `Is date:K8:7` | |
| `OTr_PutLong.4dm` | `Is real:K8:4` | All JSON numerics collapse to `Is real`; Long/Real subtypes are indistinguishable at the object level |
| `OTr_PutReal.4dm` | `Is real:K8:4` | Same caveat as PutLong |
| `OTr_PutString.4dm` | `Is text:K8:3` via `OB Get type:C1230` | Also uses `OB Is defined:C1231` |
| `OTr_PutTime.4dm` | `Is time:K8:8` | |

`OTr_PutText` delegates to `OTr_PutString` and therefore inherits the guard without change.

### 9. OTr_GetBLOB and OTr_GetNewBLOB — type-mismatch guard

Both methods were calling `OB Get` on the tag without first verifying that the stored item is actually a BLOB. Added a pre-check using `OTr_zMapType`, which correctly identifies BLOB across both storage modes (native `Is BLOB:K8:12` and base64 text with shadow key = 30):

```4d
If (OTr_zMapType($parent_o; $leafKey_t)=30)
    // retrieve BLOB via native or base64 path
Else
    OTr_zError("Type mismatch"; Current method name)
End if
```

### 11. OTr_CompareItems — cascade analysis; no code change required

The log showed OTr_CompareItems returning 0 instead of -1 for a type-mismatch scenario. Tracing the execution revealed this was not a defect in CompareItems itself but a **cascading effect**: prior Put calls without type guards had mutated shared test data, causing `OTr_zMapType` to return identical OT type values for both operands (even though the original types differed), so the "types match" branch was entered and `$result_i` was set to 0.

Once the Put type guards (item 8 above) prevented those mutations, `OTr_zMapType` returned different OT type values for the mismatched operands, the type-mismatch `Else` branch was entered, and `$result_i` remained at its initial value of -1. No change to `OTr_CompareItems.4dm` was required.

### 12. OTr_GetBoolean — type pre-check added

After the Put type guards prevented the state mutation in "textItem", `OTr_GetBoolean("textItem")` began returning 1 (True) because `OB Get` with `Is boolean` coerces non-empty text to True. OT returns 0 (False) for the same call.

This is the same pattern as `OTr_GetPicture`: the type coercion produces a misleading result for callers expecting the OT contract (empty value on mismatch). Fixed with an `OB Get type` pre-check:

```4d
If (OB Get type($parent_o; $leafKey_t)=Is boolean:K8:9)
    $value_b:=OB Get:C1224($parent_o; $leafKey_t; Is boolean:K8:9)
    If ($value_b)
        $result_i:=1
    End if
Else
    OTr_zError("Type mismatch"; Current method name)
End if
```

**File:** `Project/Sources/Methods/OTr_GetBoolean.4dm`

### 13. Phase 10c — GetString / GetText discrepancy root-cause analysis

After the Put type guards stabilised the object state, the GetString and GetText type-mismatch scenarios returned "val" (OTr) vs "" (OT). Investigation established that the test method used a single shared handle `$h_i` throughout all 120 scenarios, and the comment "scalar is Long, not String" at the GetString test site was incorrect: by that point in execution, `OTr_PutPointer` (which carries no type guard) had overwritten the BLOB-valued "scalar" with pointer-encoded text, and a subsequent `OTr_PutString` call then matched `OB Get type = Is text` and stored "val".

In OT, PutPointer on a BLOB item fails with a type mismatch; "scalar" retains its BLOB type; OT GetString on a BLOB returns "".

The root cause is test design: a single mutable shared object accumulates state across all prior scenarios. The correct fix is to **restore the known initial state** before the getter type-mismatch tests rather than adding type guards to all remaining Put methods (which would merely shift the cascade to different scenarios).

## Known Issues / Remaining Work

1. **Phase 10c GetString / GetText — pending final verification run.** The refresh block was inserted at the close of the session; the test was not re-run after that change. The refresh should result in `OTr_GetString($h_i; "scalar")` returning "" (or the coerced text representation of Long 123) rather than "val", resolving the "val" discrepancy. A short verification run is needed at the start of the next session.

2. **PutArray, PutBLOB, PutPointer, PutPicture — no type guards.** Only the six scalar Put methods received type-consistency guards this session. The array and BLOB/Pointer/Picture Put methods remain unguarded. This is the proximate cause of the shared-object cascade that produced the "val" state in the GetString test. Adding guards to these methods is deferred; the test refresh block is an adequate mitigation for Phase 10c.

3. **OTr_PutString type guard uses OB Get type, not OTr_zMapType.** The guard checks `OB Get type = Is text:K8:3`, which matches both ordinary text and pointer-encoded text (the latter has shadow key 23 but the same native type). A call to `OTr_PutString` on a Pointer-typed tag will therefore succeed, silently destroying the shadow and converting the item from Pointer to ordinary text. The correct guard would use `OTr_zMapType = 112` (OT Character). This is a known approximation; the test refresh block masks it in Phase 10c.

4. **GetString / GetText coercion on mismatch.** Like `OTr_GetVariable`, the scalar text getters coerce stored numeric values to text via `OB Get` with `Is text`. OT returns an empty string on mismatch. Adding a type pre-check (as was done for `OTr_GetBoolean`) to all scalar getters is the correct long-term fix. Deferred to a future phase.
